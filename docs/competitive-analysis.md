# 竞品分析 — pindou.org

> 版本：v0.1.0 · 最后更新：2026-06-25 · 分析对象：[pindou.org](https://pindou.org/)

## 1. 竞品定位

[pindou.org](https://pindou.org/) 是面向 perler bead 创作者的英文站（Next.js 构建），定位「图纸管理 hub」：上传图片 → 生成图纸 → 精修 → 导出，一条连贯流程。核心卖点是 **color convergence（颜色合并）** 和 **brush edits（画笔精修）**。

## 3. 借鉴清单（按价值排序）

### 🥇 P0：颜色合并（已提至 P0，F10）

- **价值**：直接决定图纸「成品感」+ 采购清单色号数（**备料成本**）。我们当前 291 色最近欧氏映射，会在色块边界和渐变区产生大量「孤豆」杂色（某色仅出现 1-3 颗）。拼豆按色号起卖（一包/一瓶），38 色里若 15 色只用到 1-3 颗 → 为 15 颗豆买 15 包，成本与库存爆炸。
- **做法**：[technical-roadmap.md](./technical-roadmap.md) 算法 3 两种模式——
  - **模式 A（空间 BFS）**：清除 1-2 颗的孤立噪点，色块边界干净。
  - **模式 B（全局归并）**：把数量少且与某大色 CIEDE2000 色差近的小色，全局归并到那个大色，色号数从几十降到 N（如 38→12）。
- **前置依赖**：补 `data/colorSystemMapping.json` 的 **LAB 字段**（见 [data-sources.md §3.3](./data-sources.md)），用 **CIEDE2000** 替换欧氏 RGB（更贴合人眼）。
- **状态**：已写入 requirements §3.1 F10（P0）、technical-roadmap 算法 3 + Phase 1。

### 🥈 画笔局部编辑（生成 → 精修）

- **价值**：图纸生成后单格不能改，用户只能整张重来。补齐「生成 → 精修」链路。
- **做法**：新增 `edit` 模式（与现有 `view` / `track` 并列）。底部图例点某色进入笔刷 → 点格子改色 / 右键擦除。`hexGrid` 已是 store 可变状态，加 `setCell(r, c, hex)` action 即可，渲染层已支持。
- **顺带实现**：色号高亮（笔刷态下选中色高亮、其余降透明度）。

### 🥉 色号高亮定位（成本极低，体验提升大）

- **价值**：点底部图例某色号 → 画布上该色格子保持原色，其余降到 ~25% 透明度，方便核对 / 连续编辑、减少出错。
- **做法**：纯渲染层，`drawGrid` 加一个 `highlightHex` 参数即可，约半小时。

### 性价比附加项

- **一键翻转**：水平 / 垂直翻转 `hexGrid`，几行代码；用户传反方向时不用重传。
- **色号搜索**：输入色号 → 高亮定位（复用高亮机制）。

## 4. 我们的护城河（不跟随）

**拍照进度识别（F15）** 是 pindou.org 完全没有的：对着半成品板拍照，diff 出错 / 缺失格子（[technical-roadmap.md](./technical-roadmap.md) 算法 5）。这是 `CLAUDE.md` 写明的核心差异化。

**策略**：借鉴 pindou.org 的「生成 + 精修」做到与其平齐，再用「进度追踪」拉开差距。

## 5. 决策记录

- **2026-06-25**：基于本分析，将「颜色合并（F10）」从 P1（AI 增强）提升至 **P0（MVP）**，作为生成质量第一优先级。理由：纯算法（无 AI 依赖，原归 P1「AI 增强」分类有误）、直接决定成品感与备料成本、竞品核心卖点。落地位置：requirements §3.1、technical-roadmap 算法 3 / Phase 1。
